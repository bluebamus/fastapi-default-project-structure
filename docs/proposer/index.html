2문제 진단 (근거 기반)

아래 6가지는 모두 “파일·라인 근거”를 동반합니다. 표면 증상이 아니라 공통 뿌리가 무엇인지에 초점을 맞췄습니다.

① “독립적인 앱” 목표와 실제의 모순 CRITICAL

새 앱을 추가하려면 중앙 파일 3곳을 반드시 수정해야 합니다. 즉 앱은 “독립”이 아니라 중앙에 강결합되어 있습니다.

수정해야 하는 중앙 파일무엇을근거
main.py라우터 include_routermain.py:229 + 232~235 주석 “향후 추가될 라우터들”
app/database/session.py모델 import + 테이블 목록 추가session.py:123–129 create_db_tables()
main.pyAdmin 뷰 add_viewmain.py:299–310
refac.md는 “신규 도메인 추가 시 기존 코드 수정 불필요”를 천명했지만, 이는 UoW에만 성립합니다. README의 “신규 모듈 개발 가이드” 8·9·10단계가 라우터·모델·Admin 중앙 등록을 그대로 요구하며 모순을 드러냅니다.

② core 인프라가 특정 앱에 역의존 HIGH

공통 미들웨어가 특정 앱 home을 import합니다. 의존성 방향이 core → app으로 역전되어, home을 떼어내면 core가 깨집니다.

# app/core/middlewares/user_info_middleware.py:15
from app.home.unit_of_work import HomeBackgroundUnitOfWork  # core가 app을 앎

③ UoW 보일러플레이트가 도메인마다 2배 HIGH

HomeUnitOfWork와 HomeBackgroundUnitOfWork는 레포지토리 구성이 완전히 동일하고 오직 세션 풀만 다릅니다. 도메인이 늘수록 거의 같은 __aenter__가 2벌씩 증식합니다.

④ Celery 미구현 + 미설치 CRITICAL

사용자 설명의 핵심(“각 앱 worker가 그 앱의 celery를 정의”)을 실현할 골격이 비어 있습니다.

  • worker/*_task.py = 전부 0줄 빈 파일
  • pyproject.toml에 celery 의존성 자체가 없음
  • 공통 Celery app · autodiscover_tasks · 태스크용 세션/UoW 기반 클래스 부재

⑤ 강제 스캐폴드의 “필수/옵션” 규약 부재 MEDIUM

앱마다 dependency.py·*_exception.py·*_admin.py·unit_of_work/·worker/가 강제되지만 대부분 빈 파일입니다. 무엇이 필수이고 무엇이 옵션인지 정의가 없어 앱마다 채움 정도가 제각각이고, 네이밍도 흔들립니다(*_task.py vs tasks.py, dependency.py vs dependencies/).

⑥ Alembic 미초기화 + 모델 수동 등록 MEDIUM

Alembic autogenerate가 모든 앱 모델을 발견하려면 모델 레지스트리가 필요합니다. 현재는 session.py의 수동 import에 의존하므로 ①의 모델 등록 문제와 같은 뿌리입니다.

공통 뿌리 한 문장 — ①②④⑥은 모두 “중앙이 앱을 일일이 알아야 한다”는 수동 등록(manual wiring)에서 비롯됩니다. 따라서 해법도 하나로 수렴합니다: 중앙이 앱을 자동으로 발견하게 만든다.

사용자가 말한 의도를 검증 가능한 설계 원칙으로 번역했습니다. 모든 제안은 이 원칙에서 도출됩니다.

4.1 폴더 재분류 & 네이밍 (확정안)

사용자 정의 (이 절의 출발점): “도메인 = 목적이 정확히 정의될 수 있는 작업의 최소 집합체. 기술/공통 = 도메인에 귀속되지 않고, 여러 도메인이 공유하는 클래스 모듈.” → 최상위 폴더 의도는 사실상 2종(도메인 / 기술·공통)이며, 별도의 “기능 앱” 범주는 두지 않습니다(home도 ‘접속로그·통계’라는 목적을 가진 도메인).

여기에 대규모(25개+)·앱 간 의존 일부 존재 조건을 결합하면 다음 확정안(domains / core / shared)으로 수렴합니다.

확정안 — domains / core / sharedapp/ ├── domains/ # 도메인 앱 = 목적이 정의된 최소 집합체 │ ├── home/ # 각 앱은 Django-app처럼 독립 │ ├── user/ blog/ reply/ sns/ │ └── commerce/order/ # (선택) 25개+ 시 바운디드 컨텍스트 하위 그룹화 │ # 레지스트리가 재귀 스캔 → 코드 변경 없이 허용 ├── core/ # 기술 인프라(도메인이 '의존'하는 공유 모듈) │ ├── registry.py # ⬅ 신규: 자동발견 핵심 │ ├── db/ # ⬅ 기존 database 통합 (session·uow·redis) │ ├── celery/ # ⬅ 신규: celery app + DBTask │ ├── models/ repositories/ services/ # Base 클래스들 │ └── exceptions/ middlewares/ └── shared/ # 도메인·프레임워크 무관 순수 유틸 └── logging/ pagination/ ...

신규/추가 이동/통합

왜 domains/가 아니라 domains/인가, 왜 core/shared를 나누는가

결정근거
domains/ 명명 (vs domains/)사용자 어휘(“도메인 = 목적의 최소 집합체”)와 1:1 일치. 폴더 이름만 봐도 “이건 도메인”임이 자명.
“기능 앱(feature)” 범주 폐기사용자 정의상 목적을 가진 단위는 전부 도메인. home도 도메인 → 별도 범주 불필요.
core ↔ shared 분리둘 다 “기술/공통”이지만 의존성 성격이 다름: core는 프레임워크에 결합돼 도메인이 의존하는 인프라, shared는 어디에도 의존 없는 순수 유틸. 분리하면 의존 방향(P2)이 폴더로 강제됨. (단일 common/으로 합치는 것도 가능 — R-6에서 선택)
25개+ 평면 + 재귀 스캔기본은 평면 domains/<name>. 규모가 커지면 domains/<context>/<name> 하위 그룹화 가능 — config.py 스캔이 재귀라 코드 변경 0.
앱 간 의존 ‘일부’ 규칙도메인→도메인 참조는 단방향·명시 선언만 허용. 양방향 공유가 필요하면 계약을 core/shared로 승격. 레지스트리가 로드 순서·순환참조를 검증.

4.2 앱 레지스트리 & 자동발견 (P1·P2의 실현)

각 앱은 AppConfig 하위 클래스 하나로 자신을 선언합니다. 선언만 하면 레지스트리가 나머지를 자동으로 연결합니다. 비활성화·순서·prefix는 명시적 탈출구로 제공합니다(P4).

① 앱이 자신을 선언한다

# app/domains/home/config.py
from app.core.registry import AppConfig

class HomeConfig(AppConfig):
    name      = "home"
    category  = "feature"        # "domain" | "feature" — 폴더가 아닌 속성
    prefix    = "/api"
    enabled   = True

    def router(self):
        from app.domains.home.api.routers.router import home_router
        return home_router

    def admin_views(self):
        from app.domains.home.admin import UserAccessLogAdmin
        return [UserAccessLogAdmin]

    # models 패키지는 관례로 자동 import → 별도 선언 불필요
    # worker/tasks.py 도 Celery autodiscover가 자동 수집

② 레지스트리가 앱을 발견·연결한다

# app/core/registry.py (요지)
class AppRegistry:
    def discover(self, package="app.domains") -> list[AppConfig]:
        # app/domains/* 하위에서 AppConfig 서브클래스를 스캔
        ...

    def import_models(self):
        # 각 앱 models 패키지 import → Base.metadata 채움 (런타임·Alembic 공용)
        ...

    def install_routers(self, app):
        for cfg in self.enabled_apps:
            if (r := cfg.router()): app.include_router(r, prefix=cfg.prefix)

    def install_admin(self, admin):
        for cfg in self.enabled_apps:
            for view in cfg.admin_views(): admin.add_view(view)

    def celery_packages(self) -> list[str]:
        return [f"app.domains.{cfg.name}" for cfg in self.enabled_apps]

③ 앱 팩토리가 부팅 시 한 번 호출한다

# app/main.py (또는 app/bootstrap.py)
def create_app() -> FastAPI:
    app = FastAPI(...)
    registry = AppRegistry()
    registry.discover()            # app/domains/* 자동 스캔
    registry.import_models()       # Base.metadata 채움
    setup_middlewares(app)         # core만 사용 (앱 비의존)
    setup_exception_handlers(app)
    registry.install_routers(app)  # ← main.py의 수동 include_router 제거
    if app_settings.ADMIN:
        registry.install_admin(Admin(app, engine))
    return app
효과(①②④⑥ 동시 해결): main.py의 수동 include_router·Admin 등록, session.py의 수동 모델 import가 모두 사라집니다. 새 앱은 domains/<name>/config.py만 추가하면 끝 — 중앙 수정 0.

4.4 UoW 보일러플레이트 제거 (P5)

도메인마다 거의 동일한 Foreground/Background 2벌을, 선언형 repositories 맵 + background 플래그로 1벌화합니다.

현재 (도메인당 2 클래스 + __aenter__ 2벌)

class HomeUnitOfWork(BaseUnitOfWork):
    user_access_logs: Repo
    async def __aenter__(self):
        await super().__aenter__()
        self.user_access_logs = Repo(self._session)
        return self

class HomeBackgroundUnitOfWork(...):  # 거의 동일 복제
    user_access_logs: Repo
    async def __aenter__(self): ...

제안 (도메인당 1 클래스, 선언만)

# app/core/db/unit_of_work.py
class BaseUnitOfWork:
    repositories: dict[str, type] = {}
    def __init__(self, session=None, *, background=False):
        self._background = background; ...
    async def __aenter__(self):
        factory = BgSession if self._background else Session
        if self._owns_session: self._session = factory()
        for attr, cls in self.repositories.items():
            setattr(self, attr, cls(self._session))
        return self

# app/domains/home/unit_of_work.py
class HomeUnitOfWork(BaseUnitOfWork):
    user_access_logs: UserAccessLogRepository       # 타입힌트
    repositories = {"user_access_logs": UserAccessLogRepository}

사용: 일반 HomeUnitOfWork(session) / 백그라운드 HomeUnitOfWork(background=True)

4.5 Alembic 모델 레지스트리 (⑥ 해결)

6전체 마이그레이션 플랜

기존 동작을 깨지 않도록 역호환 shim을 유지하며 8단계로 점진 전환합니다. 각 단계는 원자적·검증 가능·롤백 가능합니다.

gantt title 마이그레이션 단계 (의존 관계 기반 순서) dateFormat X axisFormat %s section 기반 P0 규약·레지스트리 골격(behavior无변경) :p0, 0, 1 P1 core/db 통합 + shared 분리(shim 유지) :p1, after p0, 1 section 자동발견 P2 domains/ 이전 + import 정리 :p2, after p1, 1 P3 레지스트리 연결(main/session 수동제거) :p3, after p2, 1 section 기능 P4 Celery 도입(app·autodiscover·PoC) :p4, after p3, 1 P5 Alembic init + env.py 레지스트리 :p5, after p3, 1 P6 UoW 1벌화 리팩토링 :p6, after p3, 1 section 마감 P7 생성기 + 문서 단일화(드리프트 제거) :p7, after p4, 1 P8 테스트·검증·정리 :p8, after p7, 1
  1. P0 — 규약 동결 & 레지스트리 골격
    core/registry.py, AppConfig 기반 클래스 추가(아직 미사용). 네이밍 표준 확정. 검증: import 에러 없음, 기존 테스트 그린. 롤백: 신규 파일 삭제.
  2. P1 — core/db 통합 + shared 분리
    database/ → core/db/, utils/ → shared/ 이동. 기존 경로는 re-export shim으로 유지(app/database/__init__.py가 새 위치를 re-export). 검증: 앱 부팅·테스트 그린. 롤백: shim 되돌림.
  3. P2 — domains/ 이전
    home,user,blog,reply,sns → app/domains/. import 경로는 codegraph impact로 영향 산정 후 일괄 치환. 한 앱씩 이동·검증. 검증: 앱별 엔드포인트 응답 동일. 롤백: 앱 단위 되돌림.
  4. P3 — 레지스트리 연결
    각 앱에 config.py 추가, create_app() 도입. main.py 수동 include_router·Admin, session.py 수동 모델 import 제거. core 미들웨어의 home 역의존 제거(접속로그 저장을 앱 훅/시그널 또는 옵션 미들웨어로 분리). 검증: 라우트 목록·Admin 뷰·OpenAPI 동일. 롤백: 수동 등록 복원.
  5. P4 — Celery 도입
    celery 의존성 추가, core/celery/app.py + DBTask + autodiscover. async UoW 브리지 PoC 후 home에 실제 태스크 1개 구현(접속로그 집계 등). 검증: 워커가 태스크 발견·실행, DB 반영. 롤백: celery 진입점 비활성.
  6. P5 — Alembic 초기화
    alembic init migrations, env.py가 AppRegistry().import_models() 사용. 현 스키마 기준 baseline 마이그레이션 생성. 검증: autogenerate가 전체 모델 인식, upgrade/downgrade 동작. 롤백: migrations 폐기.
  7. P6 — UoW 1벌화
    BaseUnitOfWork에 선언형 repositories + background 플래그 도입. 도메인 *BackgroundUnitOfWork 제거, 호출부를 background=True로 치환. 검증: 트랜잭션·롤백·풀 분리 동작 동일. 롤백: 2벌 구조 복원.
  8. P7 — 생성기 & 문서 단일화
    scripts/new_app.py 추가. README/refac.md/flow.md를 1개의 살아있는 ARCHITECTURE 문서로 통합, 드리프트 제거(P6 SSoT). 검증: 생성기로 만든 앱이 무수정 부팅. 롤백: 문서 되돌림.
  9. P8 — 테스트·검증·정리
    shim 제거, 빈 옵션 파일 정리, 전 앱 스모크 테스트. verification-before-completion 기준 통과. 검증: 전체 테스트·부팅·OpenAPI·Admin·Celery·Alembic 그린.
안전장치: P1·P2의 re-export shim 덕분에 각 단계가 독립 PR로 머지 가능하며, 어느 단계에서 멈춰도 앱은 항상 동작합니다. P4·P5·P6은 P3 이후 병렬 진행이 가능합니다.
