1세대 자동발견(Auto-Discovery) 레지스트리 main 제거 → 브랜치 복원

이 프로젝트는 한때 Django식 "앱 자동발견" 구조를 갖고 있었습니다. 각 도메인 앱이 스스로를 선언하면, 부팅 시 중앙 레지스트리가 app/domains/ 아래를 재귀 스캔해서 라우터·모델·Admin·Celery 태스크를 자동으로 연결하는 방식이었습니다. 즉 새 앱을 추가할 때 중앙 파일을 한 줄도 고칠 필요가 없는 설계였습니다.

이후 이 자동발견은 명시적 수동등록(app/apps.py)으로 교체되었습니다(전환 분석은 자매 문서 app-registration-installed-apps-2026-06-25.html 참고). 이 문서는 사라진 1세대가 정확히 어떻게 설계·동작했는지를 도식과 함께 복원·정리합니다.

한 문장 요약. 두 개의 클래스 AppConfig(각 도메인의 자기선언)와 AppRegistry(스캐너)가 핵심이었고, pkgutil.walk_packages로 도메인 트리를 훑어 config.py 안의 AppConfig 하위 클래스를 모아 4개 소비처(웹·Celery·DB·Alembic)에 흘려보냈습니다.

0. 먼저 — 핵심 용어 정의

이 문서를 편하게 읽기 위한 5개 용어입니다.

자동발견(Auto-Discovery). 개발자가 "이 앱을 켜라"라고 어딘가에 적지 않아도, 프로그램이 실행 시점에 디렉토리를 뒤져 앱을 스스로 찾아내는 방식. 반대말은 "수동 등록(명시적 목록에 직접 추가)".
레지스트리(Registry). 발견한 앱들을 모아두고 "라우터 설치", "모델 import" 같은 작업을 대신 실행해 주는 중앙 객체. 여기서는 AppRegistry 클래스.
AppConfig. 각 도메인 앱이 "나는 이런 앱이고, 내 라우터·Admin은 여기 있다"라고 적어두는 자기소개 클래스. Django의 apps.py 안 AppConfig와 같은 역할. 도메인의 config.py에 들어 있었습니다.
훅(Hook) 메서드. 베이스가 비워둔 채 "필요하면 하위 클래스가 채워라"라고 열어 둔 메서드. 예: router()는 기본값이 None이고, 라우터가 있는 앱만 오버라이드.
소비처(Consumer). 레지스트리가 모은 정보를 실제로 사용하는 코드. 1세대에는 4곳이 있었습니다 — 웹 부트스트랩, Celery 앱, DB 테이블 생성, Alembic 마이그레이션.

1. 큰 그림 — 무엇이 무엇과 연결되었나

세 층으로 보면 쉽습니다. (아래) 도메인 앱들이 각자 config.py로 자기를 선언하고 → (가운데) 레지스트리가 그것들을 스캔·수집하고 → (위) 소비처들이 레지스트리에 물어 필요한 것을 가져갑니다. 도메인은 소비처를 모르고, 소비처도 도메인을 직접 모릅니다 — 오직 레지스트리를 통해서만 만납니다.

2. AppConfig — 도메인의 "자기소개서"

모든 도메인 앱은 app/core/registry.py의 AppConfig를 상속해, 자기 디렉토리의 config.py에 설정 클래스를 하나 둡니다. 클래스 속성은 "정적 사실"(이름·prefix·로드 순서)이고, 메서드는 "필요할 때 꺼내 주는 것"(라우터·Admin·스케줄)입니다. 

# app/core/registry.py (복원) — 베이스 클래스
class AppConfig:
    """도메인 앱의 자기 선언. 하위 클래스가 클래스 속성/훅을 오버라이드한다."""

    name: str = ""            # 앱 고유 이름 (중복 금지)
    category: str = "domain"  # 분류 (거의 항상 domain)
    prefix: str = "/api"      # 이 앱 라우터의 URL 프리픽스
    enabled: bool = True      # False면 발견돼도 건너뜀 (스위치)
    order: int = 100          # 낮을수록 먼저 로드 (앱 간 의존 순서 제어)

    @property
    def package(self) -> str:
        # type(self).__module__ == "app.domains.home.config"
        # → 마지막 ".config"를 떼어 "app.domains.home" 을 돌려준다
        module = type(self).__module__
        return module.rsplit(".", 1)[0]

    def router(self) -> APIRouter | None: return None   # 훅: 라우터 있으면 오버라이드
    def admin_views(self) -> list[type]: return []      # 훅: Admin 뷰
    def beat_schedule(self) -> dict: return {}          # 훅: Celery 스케줄 조각
구성종류의미 / 사용처
name속성앱 식별자. 발견 단계에서 중복되면 부팅 실패(RuntimeError).
prefix속성라우터 마운트 시 URL 접두어. 기본 /api.
enabled속성False면 발견돼도 제외 — Django의 INSTALLED_APPS에서 빼는 것과 유사한 "끄기 스위치".
order속성로드 순서. 낮을수록 먼저. 앱 간 의존(예: 공통 앱 먼저)을 제어.
package프로퍼티모듈 경로에서 패키지를 역산. 모델·태스크 경로의 기준점({package}.models 등).
router()훅앱의 통합 APIRouter. 없으면 None(웹 라우터 없는 앱도 허용).
admin_views()훅SQLAdmin ModelView 목록.
beat_schedule()훅Celery beat 스케줄 조각. 레지스트리가 전부 병합.

실제 예시 — home 앱의 자기소개서

# app/domains/home/config.py (복원)
from fastapi import APIRouter
from app.core.registry import AppConfig
from app.core.middlewares.access_log_sink import set_access_log_sink
from app.domains.home.access_log_sink import HomeAccessLogSink

set_access_log_sink(HomeAccessLogSink())   # ⚠️ import 시점 부수효과 (9장에서 다룸)


class HomeConfig(AppConfig):
    name = "home"
    category = "domain"
    prefix = "/api"
    order = 10                              # home을 먼저 로드

    def router(self) -> APIRouter:
        from app.domains.home.api.routers.router import home_router  # 지연 import
        return home_router

    def admin_views(self) -> list[type]:
        from app.domains.home.admin import UserAccessLogAdmin
        return [UserAccessLogAdmin]

메서드 안에서 import를 하는 "지연 import"에 주목하세요. config.py가 로드될 때가 아니라, 레지스트리가 실제로 router()를 호출할 때 라우터를 가져옵니다 — 순환참조와 불필요한 조기 로딩을 피하는 장치입니다.

3. AppRegistry — 스캐너이자 설치 기사

AppRegistry는 두 가지 일을 합니다. ① 발견(discover) — 도메인 트리를 훑어 AppConfig들을 모은다. ② 설치(install_*) — 모은 것들을 소비처가 준 대상(앱·Admin)에 꽂아 준다. 발견 결과는 내부 _apps 리스트에 보관됩니다.

그림 2. 클래스 관계 — 도메인의 HomeConfig는 AppConfig를 상속하고, AppRegistry는 그런 config들을 모아 보관·설치한다.

설치 메서드들 — 모은 것을 어디에 꽂는가

# app/core/registry.py (복원) — 설치 메서드
def install_routers(self, app) -> int:          # 웹 앱에 라우터 마운트
    count = 0
    for cfg in self._apps:
        router = cfg.router()
        if router is not None:
            app.include_router(router, prefix=cfg.prefix)
            count += 1
    return count

def import_models(self) -> None:                # 각 앱의 models 모듈 import → Base.metadata 채움
    for cfg in self._apps:
        try:
            importlib.import_module(f"{cfg.package}.models")
        except ModuleNotFoundError:
            continue                            # 모델 없는 앱은 조용히 통과

def install_admin(self, admin) -> int:          # SQLAdmin에 뷰 등록
    count = 0
    for cfg in self._apps:
        for view in cfg.admin_views():
            admin.add_view(view)
            count += 1
    return count

def celery_packages(self) -> list[str]:         # Celery autodiscover 대상 패키지 목록
    return [cfg.package for cfg in self._apps]

def merged_beat_schedule(self) -> dict:         # 모든 앱의 스케줄 조각 병합
    schedule: dict = {}
    for cfg in self._apps:
        schedule.update(cfg.beat_schedule())
    return schedule

설치 메서드들은 하나같이 "self._apps를 순회하며 각 config의 훅을 호출"하는 단순 루프입니다. 복잡함은 전부 discover() 한 곳에 모여 있습니다 — 그것이 다음 장입니다.

4. discover()의 내부 — 자동발견의 심장

자동발견의 "마법"은 전부 이 메서드 안에 있습니다. 말로 풀면 이렇습니다: "app.domains 패키지 아래 모든 하위 모듈을 걸어다니다가, 이름이 .config로 끝나는 모듈을 만나면 import해서, 그 안에 정의된 AppConfig의 진짜 하위 클래스를 꺼내 인스턴스로 만든다. 켜져 있고 이름이 안 겹치면 채택하고, 마지막에 order 순으로 줄 세운다." 

# app/core/registry.py (복원) — 발견 엔진
def discover(self, package: str = "app.domains") -> list[AppConfig]:
    root = importlib.import_module(package)
    found: dict[str, AppConfig] = {}

    # ① app.domains 아래 모든 하위 패키지/모듈을 재귀적으로 순회
    for mod_info in pkgutil.walk_packages(root.__path__, prefix=f"{package}."):
        # ② 이름이 ".config" 로 끝나는 모듈만 관심 대상
        if not mod_info.name.endswith(".config"):
            continue
        module = importlib.import_module(mod_info.name)        # ③ 그 config 모듈 import (부수효과 실행됨)
        # ④ 모듈 안의 모든 클래스 중 AppConfig 의 '직접 정의된' 하위 클래스만
        for _, obj in inspect.getmembers(module, inspect.isclass):
            if (issubclass(obj, AppConfig) and obj is not AppConfig
                    and obj.__module__ == mod_info.name):       # ← import해 온 클래스 제외
                cfg = obj()                                     # ⑤ 인스턴스화
                if not cfg.enabled:                             # ⑥ 꺼진 앱 건너뜀
                    continue
                if cfg.name in found:                           # ⑦ 이름 충돌 → 즉시 실패
                    raise RuntimeError(f"중복된 앱 이름: {cfg.name}")
                found[cfg.name] = cfg

    # ⑧ order, name 순 정렬 후 보관
    self._apps = sorted(found.values(), key=lambda c: (c.order, c.name))
    return self._apps

그림 3. discover() 알고리즘 — 재귀 순회 → .config 필터 → 클래스 판별 → 활성/중복 검사 → 정렬.

왜 이런 조건들이 필요했나 — 세 가지 "안전장치"

  • obj.__module__ == mod_info.name: config.py는 보통 from app.core.registry import AppConfig로 베이스를 import합니다. 이 조건이 없으면 import해 온 AppConfig까지 "클래스"로 잡혀 버립니다. "이 모듈에서 실제로 정의된 클래스만"이라는 필터입니다.
  • obj is not AppConfig: 베이스 자신은 앱이 아니므로 제외.
  • 이름 중복 → RuntimeError: 두 앱이 같은 name을 쓰면 라우터/모델이 덮어써져 조용히 깨지는 대신, 부팅을 즉시 중단해 개발자에게 알립니다("실패를 빠르게").
핵심 부작용(③). discover()는 발견을 위해 모든 config.py를 import합니다. 그래서 home/config.py 맨 위의 set_access_log_sink(HomeAccessLogSink()) 같은 모듈 레벨 코드가 "발견되는 순간" 실행됩니다. 이것이 1세대의 강력함이자 동시에 9장에서 다룰 위험의 근원입니다.

    registry.import_models()       # ② Base.metadata 채움

    app = FastAPI(...)
    # ... 미들웨어/예외핸들러 ...
    n = registry.install_routers(app)          # ③ 라우터 마운트

    _add_health_and_docs(app)

    if app_settings.ADMIN:

        registry.install_admin(admin)          # ④ Admin 뷰 등록

Celery의 autodiscover_tasks(..., related_name="worker.tasks")는 "각 패키지의 worker/tasks.py를 찾아 태스크를 등록하라"는 뜻입니다. 즉 1세대는 라우터·모델뿐 아니라 Celery 태스크까지 자동발견했습니다.

7. 손으로 따라가는 예시 — home이 등록되는 전 과정

추상적 설명을 구체적인 한 줄짜리 여정으로 바꿔 봅시다. 웹 서버가 막 시작했습니다.

  1. 1create_app()이 registry.discover()를 호출.
  2. 2레지스트리가 app.domains를 import하고 walk_packages로 하위를 순회하다 app.domains.home.config를 만남(이름이 .config로 끝남 ✔).
  3. 3그 모듈을 import → 맨 윗줄 set_access_log_sink(HomeAccessLogSink())가 이 순간 실행됨.
  4. 4모듈 안에서 HomeConfig 발견 — AppConfig의 하위이고, __module__이 그 모듈과 일치 ✔.
  5. 5HomeConfig() 인스턴스화. enabled=True ✔, name="home" 중복 없음 ✔ → 채택.
  6. 6더 이상 config 없음 → order 순 정렬(home=10) → _apps = [HomeConfig()].
  7. 7import_models(): cfg.package가 "app.domains.home"이므로 app.domains.home.models를 import → user_access_logs 테이블이 Base.metadata에 등록.
  8. 8install_routers(app): HomeConfig.router()가 home_router를 반환 → app.include_router(home_router, prefix="/api") → /api/v1/home/... 경로가 살아남.
  9. 9(ADMIN=True면) install_admin(admin): UserAccessLogAdmin이 /admin에 등록.
핵심 포인트. 이 9단계 어디에도 "home을 켜라"고 적은 중앙 코드가 없습니다. home/config.py 파일이 존재한다는 사실 자체가 등록의 트리거였습니다.

8. 새 앱 추가 — "중앙 편집 0회"의 의미

1세대에서 orders 앱을 추가한다면 절차는 이게 전부였습니다:

  1. 스캐폴딩으로 app/domains/orders/ 디렉토리 생성.
  2. app/domains/orders/config.py에 OrdersConfig(AppConfig)를 작성: 
    class OrdersConfig(AppConfig):
    name = "orders"
    order = 50
    def router(self):
        from app.domains.orders.api.routers.router import orders_router
        return orders_router
  3. 끝. bootstrap.py·celery/app.py·migrations/env.py를 건드리지 않습니다. 다음 부팅 때 discover()가 자동으로 orders.config를 주워 라우터·모델·태스크를 연결합니다.
현재(2세대)와의 대비. 지금은 같은 일을 하려면 app/apps.py의 routers()·_MODEL_MODULES 등에 직접 항목을 추가해야 합니다. 1세대는 "파일을 두면 켜진다", 2세대는 "목록에 적어야 켜진다"의 차이입니다.

9. 왜 제거되었나 — 자동발견의 그림자

자동발견은 편리했지만, 전환 커밋 07e73b1은 이를 의도적으로 걷어냈습니다. 정직하게 말하면 1세대에는 다음과 같은 구조적 비용이 있었습니다.

그림자설명
import 시점 부수효과 discover()가 config.py를 import하는 순간 set_access_log_sink(...) 같은 모듈 레벨 코드가 실행됨. "발견"과 "부수효과 실행"이 얽혀, 언제 무엇이 실행되는지 추적이 어려움.
스캔 마법(암묵성) 활성 앱 목록이 코드 어디에도 명시되지 않음. "지금 무엇이 켜져 있나?"를 알려면 런타임 스캔 결과를 봐야 함.
중복 스캔 웹·Celery·Alembic이 각자 AppRegistry()를 만들어 discover()를 호출 — 같은 작업을 3번 반복.
암묵적 순서 의존 앱 간 로드 순서를 order 정수로 제어 — 충돌·의존이 생기면 숫자 조정이라는 미묘한 메커니즘에 의존.
디버깅 난이도 pkgutil/inspect 기반 동적 발견은 IDE·정적분석이 따라가지 못해, "왜 이 앱이 안 떴지?"의 추적이 어려움.
