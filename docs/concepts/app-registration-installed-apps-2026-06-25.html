FastAPI의 INSTALLED_APPS 유사 기능 — 구현·동작 추적

Django에는 활성 앱을 한곳에 선언하는 INSTALLED_APPS가 있지만 FastAPI에는 그런 내장 개념이 없습니다. 이 프로젝트는 이전 요청에 따라 Django식 앱 등록 구조를 모사했고, 그 결과물이 app/apps.py(앱 수동 등록 SSOT)입니다. 이 문서는 그 구현 내용과 실제 동작 여부를 추적·정리합니다.

결론 요약. Django INSTALLED_APPS의 역할은 app/apps.py 단일 모듈로 구현되어 있으며, 4개 소비처(부트스트랩·Celery·DB세션·Alembic)에 연결되어 실제로 동작합니다 (등록 테스트 5/5 통과 + 앱 부팅·라우트 마운트·메타데이터 등록 실측 확인). 단, 현재 활성 앱은 home 하나뿐이며 blog·reply·sns·user는 디스크에만 존재하고 미등록 상태(=비활성)입니다.

1. 출발점 — Django INSTALLED_APPS vs FastAPI

Django는 프로젝트 부팅 시 settings.INSTALLED_APPS에 나열된 앱들을 스캔하여 모델·admin·관리 명령·시그널 등을 자동 등록합니다. 핵심 속성은 두 가지입니다.

  • 단일 선언 지점: "어떤 앱이 활성인가"가 한 리스트에 모인다.
  • 디스크 존재 ≠ 활성: 앱 코드가 있어도 리스트에 없으면 로드되지 않는다.

FastAPI에는 이런 메커니즘이 없습니다. 라우터·모델·태스크를 개발자가 직접 include_router() 등으로 붙여야 합니다. 이 프로젝트는 그 "직접 붙이는 일"을 한 파일에 모아 SSOT(Single Source of Truth)로 만든 것이 INSTALLED_APPS에 대응하는 설계입니다.

2. 진화 추적 — 자동발견 → 명시적 수동등록

이 기능은 두 세대를 거쳤습니다. git 이력으로 확인된 전환점은 커밋 07e73b1입니다.

세대구현방식상태
1세대 (이전) app/core/registry.py — AppConfig/AppRegistry
+ 도메인별 config.py		 Django 모사 자동 발견(pkgutil/inspect로 도메인 패키지 스캔). Django AppConfig를 가장 가깝게 흉내. 커밋 07e73b1에서 삭제됨
2세대 (main 현행) app/apps.py 단일 SSOT 명시적 수동 등록. 스캔 없음. 활성 앱을 리스트/함수에 직접 적는다. main 현행
3세대 (브랜치 현행) app/core/registry.py 복원 + 도메인별 config.py 자동발견 재도입. app/apps.py 제거. config.py 만 두면 자동 등록(1세대 메커니즘 복원). feature/auto-app-discovery

전환 커밋의 메시지(요지):

refactor(core): replace auto-discovery with explicit manual app registration

Remove the Django-style AppConfig/AppRegistry auto-scan (pkgutil/inspect) and register domain apps explicitly in a single SSOT: app/apps.py. … Delete app/core/registry.py and app/domains/home/config.py … Rewire 4 consumers (bootstrap, celery/app, db/session, migrations/env) to import from app.apps instead of running discover() … Celery: drop autodiscover_tasks; use explicit include=CELERY_TASK_MODULES.

즉 "Django식 자동 마법"을 의도적으로 걷어내고, Django INSTALLED_APPS의 명시성만 취한 형태입니다. 자동 스캔(1세대)이 INSTALLED_APPS 동작에 더 가까웠고, 현재(2세대)는 그 선언 리스트 측면을 계승합니다.

3. 현재 구현 — app/apps.py (SSOT)

파일 app/apps.py는 네 가지 등록 축을 노출합니다. 각각이 Django 앱 시스템의 한 측면에 대응합니다.

심볼형태등록 대상현재 값
routers()함수 → list[RouterSpec]FastAPI 라우터(+prefix)[home_router @ /api]
_MODEL_MODULES + register_models()리스트 + 함수SQLAlchemy 모델 → Base.metadata["app.domains.home.models.models"]
admin_views()함수 → list[type]SQLAdmin ModelView[UserAccessLogAdmin]
CELERY_TASK_MODULES리스트Celery 태스크 모듈["app.domains.home.worker.tasks"]
BEAT_SCHEDULEdictCelery beat 스케줄{}

핵심 설계 포인트:

  • 지연 import: routers()는 함수 내부에서 라우터를 import한다 — 모듈 로드 시점의 부수효과·순환참조를 피하기 위함.
  • 등록 훅 보존: routers()는 home 라우터를 반환하기 전에 access_log_sink.register_sink()를 명시 호출한다(1세대 config.py 부수효과를 대체). → "core는 도메인을 모른다" 규칙 유지.
  • 모델 등록 = import 부수효과: register_models()가 모델 모듈을 import하면 클래스 정의가 Base.metadata에 채워진다(테이블 생성·Alembic 공용).
# app/apps.py (발췌)
@dataclass(frozen=True)
class RouterSpec:
    router: APIRouter
    prefix: str = "/api"

def routers() -> list[RouterSpec]:
    from app.domains.home.access_log_sink import register_sink
    register_sink()                                   # 등록 훅 (core↔domain 분리 유지)
    from app.domains.home.api.routers.router import home_router
    return [RouterSpec(router=home_router, prefix="/api")]

_MODEL_MODULES = ["app.domains.home.models.models"]
def register_models() -> None:
    for module_path in _MODEL_MODULES:
        importlib.import_module(module_path)          # import → Base.metadata 채움

def admin_views() -> list[type]:
    from app.domains.home.admin import UserAccessLogAdmin
    return [UserAccessLogAdmin]

CELERY_TASK_MODULES = ["app.domains.home.worker.tasks"]
BEAT_SCHEDULE: dict = {}

7. 주목할 발견 & 설계 트레이드오프

발견 — 디스크에 있어도 미등록이면 비활성. app/domains/ 아래에는 home 외에도 blog · reply · sns · user 도메인이 라우터·모델 파일과 함께 존재합니다. 하지만 app/apps.py에는 home만 등록되어 있어 나머지는 마운트되지 않습니다(라우트·테이블에 나타나지 않음).

이는 버그가 아니라 Django INSTALLED_APPS와 정확히 동일한 의미론입니다 — 앱 코드가 저장소에 있어도 등록 리스트에 없으면 비활성. 명시적 등록 철학의 자연스러운 귀결입니다. (이 도메인들은 표준 레이아웃을 보여주는 스캐폴드/예시로 남아 있습니다.)

그림 5. "디스크 존재 ≠ 활성." 5개 도메인이 모두 라우터·모델 파일을 갖지만, app/apps.py에 적힌 home만 실제로 살아난다. §6 실측(TABLES: ['user_access_logs'])이 이를 뒷받침.
